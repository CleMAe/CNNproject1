# CNNProject1

基于铸件图像缺陷检测任务的端到端项目。项目从数据解压、中文 EDA、EfficientNet 建模训练、评估与推理，一直扩展到 `FastAPI + Docker` 的 Web 服务部署，便于后续通过 HTTP 接口对接业务系统。

## 项目亮点

- 完成数据解压、数据质量检查、分布分析与中文可视化 EDA
- 构建基于 EfficientNet 的缺陷分类模型，支持训练、评估、推理
- 针对类别不平衡使用 `WeightedRandomSampler + FocalLoss`
- 输出完整评估结果，包括分类报告、混淆矩阵、ROC、PR 曲线
- 将模型封装成 `FastAPI` 服务，返回 JSON，适合接口化部署
- 完成 Docker 容器化部署，服务端口固定为 `9053`
- 支持接口压测模拟，可生成每分钟请求运行曲线

## 当前仓库内容

- `scripts/run_eda.py`：中文 EDA 主脚本
- `scripts/train_efficientnet.py`：EfficientNet 训练脚本
- `scripts/evaluate_efficientnet.py`：模型评估脚本
- `scripts/infer_efficientnet.py`：单图推理脚本
- `scripts/simulate_fastapi_requests.py`：HTTP 接口模拟访问脚本
- `src/cnnproject1/`：模型、数据集、推理服务核心实现
- `app/main.py`：FastAPI 应用入口
- `app/templates/index.html`：可视化首页
- `app/static/styles.css`：前端样式
- `app/static/app.js`：前端交互逻辑
- `reports/`：EDA 报告与图表

## Web 服务使用

本地启动：

```bash
uvicorn app.main:app --host 0.0.0.0 --port 9053
```

访问首页：

```bash
http://127.0.0.1:9053/
```

访问 Swagger：

```bash
http://127.0.0.1:9053/docs
```

健康检查：

```bash
curl http://127.0.0.1:9053/health
```

图片检测接口：

```bash
curl -X POST "http://127.0.0.1:9053/predict" \
  -F "file=@你的图片路径"
```

接口返回值为 JSON，包含：

- 请求编号
- 文件名
- 预测类别
- 预测置信度
- 各类别概率
- 推理耗时
- 原始图片尺寸
- 模型名称、设备、检查点路径

## Docker 部署

构建镜像：

```bash
docker build -t cnnproject1-fastapi .
```

运行容器：

```bash
docker run -d --name cnnproject1-fastapi-app -p 9053:9053 cnnproject1-fastapi
```

查看日志：

```bash
docker logs -f cnnproject1-fastapi-app
```

## 接口压测

按“每分钟 20 个请求，累计 10 分钟”模拟调用：

```bash
python3 scripts/simulate_fastapi_requests.py \
  --base-url http://127.0.0.1:9053 \
  --requests-per-minute 20 \
  --duration-minutes 10 \
  --output-dir web_outputs/load_test
```

输出结果包括：

- `接口请求明细.csv`
- `接口压测摘要.json`
- `接口运行曲线.png`

## 建模说明

- 默认根据 EDA 结果过滤训练集和测试集之间的重复图片，降低数据泄漏风险
- 使用 `EfficientNet` 完成二分类建模
- 使用 `WeightedRandomSampler + FocalLoss` 缓解类别不平衡
- 支持本地模型加载推理，满足离线部署需求
- 已将 GAN 相关探索视为独立试验，不纳入当前主线系统

## 基于 STAR 法则的 3 分钟面试口语稿

下面这段可以直接用于面试口述：

```text
这个项目是一个铸件缺陷检测系统，我把它从一个单纯的数据建模任务，完整推进成了一个可以通过 HTTP 调用的部署型项目。

先说背景，也就是 Situation。当时项目的目标是基于铸件图像识别产品是否存在缺陷，但最开始只是一个原始数据集，没有成熟的数据分析流程，也没有真正可交付的服务接口。如果只停留在训练模型层面，后续很难真正接入业务系统。

然后是 Task。我的任务不只是把模型训出来，而是要做一整套闭环，包括数据解压、EDA、建模训练、评估、推理、Web 化封装、接口压测和 Docker 部署。换句话说，我要把它做成一个既能研究、又能落地的项目。

接下来是 Action。第一步，我先做了完整的中文 EDA，重点分析了数据质量、类别分布、图像尺寸、亮度、清晰度以及重复样本，还识别出了训练集和测试集之间的重复图片，用来规避数据泄漏。第二步，在建模阶段我选择了 EfficientNet，并结合 WeightedRandomSampler 和 FocalLoss 去处理类别不平衡问题，同时补齐了训练曲线、分类报告、混淆矩阵、ROC 和 PR 曲线，让评估更完整。第三步，我把本地推理能力抽成服务层，用 FastAPI 暴露成 `/health` 和 `/predict` 两个接口，返回统一 JSON，方便后续前后端或第三方系统直接调用。第四步，我又补了一个更直观的前端页面，支持图片上传、结果展示、概率条可视化和 JSON 结果查看。最后，我完成了 Docker 容器化，并设计了每分钟 20 次请求、持续 10 分钟的接口模拟访问脚本，能输出运行曲线，验证服务在持续调用下的表现。

最后是 Result。这个项目最终不只是一个分类模型，而是一个可以直接部署和演示的缺陷检测系统。它既有算法部分，也有工程化部分。我在里面体现出来的能力，主要包括数据分析能力、PyTorch 建模能力、接口化改造能力，以及把模型真正包装成交付系统的能力。如果放到业务场景里，这种形态会更容易接入产线质检、MES 系统或者内部质量看板。
```
